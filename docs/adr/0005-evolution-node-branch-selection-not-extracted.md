# buildEvolutionNode's priority-chain decision stays inline

An architecture review flagged `buildEvolutionNode`'s branch-selection decision (`normalize.ts:282-310` — the `viewedDetail > exactDetail > dedicatedSuffixes-drop > pluralize > unconditional > default-drop` priority chain) as lacking its own test locality: pure logic, only reachable through full raw-chain fixtures via `normalizeEvolutionChain`, never unit-tested on its own.

On inspection this understates existing coverage: `normalize.test.ts` already has 20+ targeted tests against this exact priority chain, each using a small hand-built `RawEvolutionChainLink` fixture (not a full 900-species raw chain) exercised through the public `normalizeEvolutionChain` entry point — e.g. `normalize.test.ts:405` tests the "drop when viewing the wrong form" branch, `:410` tests the exact-form-match branch, `:366`/`:371` test the dedicated-suffix-drop branch. This is already per-branch locality, just expressed as small realistic fixtures rather than a hand-fed decision-table unit test.

More importantly, this codebase's actual historical bugs in this area (Obstagoon's undercounted evolution-family depth, the Corsola/Mime-Jr chain fixes — see CLAUDE.md) were all in *context propagation across the recursion* (`inferAncestorFormSuffix`, `targetSuffix` carrying across levels), not in the priority-chain's own ordering. Extracting just the if/else into a standalone function and unit-testing it with hand-fed `viewedDetail`/`exactDetail` stand-ins would test the ranking in isolation, disconnected from the derivation logic that has actually produced bugs — it would not have caught any of them.

**Decision:** leave the priority-chain decision inline in `buildEvolutionNode`. Don't re-flag it as a locality/testability gap in a future review without first checking `normalize.test.ts` for existing branch-level coverage — the check is "does a small-fixture test already exercise this branch," not "is there a standalone unit test for the extracted decision function."
